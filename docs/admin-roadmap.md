Micro-Task Map — Admin Dashboard
--------------------------------

| #   | Task                                                                                                      | Why it's sequenced here                                                                                                                         |
| --- | --------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Environment & Supabase clients — `.env.local`, `.env.example`, `src/lib/supabase/client.ts` + `server.ts` | Nothing can talk to the DB until credentials and the official Supabase SSR clients exist; this prevents wiring UI to a broken connection later. |
| 2   | Shared DB types — `src/types/database.ts` (Product, Order, Profile, Review shapes)                        | Typed data flows prevent silent mismatches with `database_schema.sql` before any page fetches rows.                                             |
| 3   | Connection smoke test — minimal Server Component or action that reads product count from Supabase         | Proves env + clients + RLS read path work before building admin UI on top.                                                                      |
| 3.5 | Design System Setup (Tailwind v4)                                                                         |                                                                                                                     |
| 4   | Admin layout shell — `src/app/admin/layout.tsx` with RTL nav (Products, Orders, Reviews, Dashboard)       | Stable UI chrome first; pages plug into a consistent shell instead of one-off layouts.                                                          |
| 5   | Admin middleware — `src/middleware.ts` protecting `/admin/*` (except `/admin/login`)                      | Route guard before CRUD so admin routes aren't briefly public while pages are built.                                                            |
| 6   | Admin login page — `/admin/login` with Email/Password sign-in                                             | Auth must work before role-checked CRUD; email first keeps OAuth as a later increment.                                                          |
| 7   | Admin dashboard page — `/admin` with 3 metric placeholders (sales, pending orders, active products)       | Lightweight landing page validates layout + auth redirect loop before heavy CRUD.                                                               |
| 8   | Products list (read-only) — `/admin/products` table fetching all products (including inactive)            | Read before write: confirm admin SELECT + RLS + table rendering before mutations.                                                               |
| 9   | Product create — add form + Server Action (title, slug, description, price, stock, category, `is_active`) | Single create path validates Zod + Server Action pattern before edit/delete.                                                                    |
| 10  | Product edit — pre-filled form + update Server Action                                                     | Reuses create patterns; confirms slug uniqueness and update RLS.                                                                                |
| 11  | Product soft-delete toggle — `is_active` toggle in the list                                               | Matches architecture (no hard delete); quick win before image upload complexity.                                                                |
| 12  | Product image upload — Supabase Storage upload + save URLs in `products.images[]`                         | Storage is a separate integration; deferred until text CRUD is stable.                                                                          |
| 13  | Orders list — `/admin/orders` table (customer, phone, address, total, status)                             | Operational core — admin must see incoming orders after products are manageable.                                                                |
| 14  | Order detail view — expand row or detail panel with `order_items` line items                              | Fulfillment needs line-item snapshots, not just order headers.                                                                                  |
| 15  | Order status update — Server Action: `pending` → `shipped` → `completed` (and `cancelled`)                | Closes the admin order lifecycle that the business runs on daily.                                                                               |
| 16  | Reviews list — `/admin/reviews` showing published + hidden reviews                                        | Read path before manual entry form.                                                                                                             |
| 17  | Review create — admin form + Server Action (name, text, rating 1–5, linked product)                       | Last admin feature; reviews are secondary to products and orders.                                                                               |

18 tasks · ~10–15 min each · each ends with one manual test + Git commit before the next.
